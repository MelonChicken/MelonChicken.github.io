## <span>무엇을 만드는가</span>

> <span>깃허브 링크: </span><a href={"https://github.com/MelonChicken/be-more-agent-duck"}><span>https://github.com/MelonChicken/be-more-agent-duck</span></a>

<span><strong>BMOD</strong></span><span>는 오리 영상을 입력받아 행동을 분류하고, 행동 변화에 반응하며, 세션이 끝나면 리포트를 생성하는 </span><span><strong>lightweight behaviour observation system</strong></span><span>입니다.</span>

<span>파이프라인을 한 줄로 정리하면 이렇습니다:</span>

```plaintext
입력 영상
  → 5초 구간 분할
  → 구간당 10프레임 샘플링
  → CLIP 임베딩 평균
  → Logistic Regression 분류
  → 이벤트 감지
  → 반응 생성
  → 세션 리포트 (CSV + JSON)
```

<span>행동 클래스는 세 가지입니다: </span><span><strong>resting</strong></span><span>, </span><span><strong>feeding</strong></span><span>, </span><span><strong>exploring</strong></span><span>. 그리고 분류 신뢰도가 낮을 때를 위한 </span><span><strong>uncertain</strong></span><span> 상태도 따로 두었습니다.</span>

### <span>왜 이 스택인가</span>

<span>처음엔 성능만 따지면 훨씬 좋은 end-to-end video model도 고려했습니다. 그런데 이 프로젝트의 최종 목표는 </span><span><strong>Raspberry Pi 5 위에서 오프라인으로 돌아가는 것</strong></span><span>이었기에 무거운 모델은 선택지가 될 수 없었습니다.</span>

<span>CLIP 임베딩 + Logistic Regression 조합은 다음 이유로 선택했습니다:</span>

- <span>Pi에 이식하기 충분히 가볍다</span>
- <span>노트북에서 개발하고 Pi로 옮기기 쉽다</span>
- <span>행동 분류에 필요한 visual feature를 CLIP이 이미 잘 뽑아준다</span>

<span>물론 리스크도 있습니다. CLIP feature가 행동 자체보다 </span><span><strong>장면 특성</strong></span><span> (배경, 조명 등)에 반응할 수 있다는 점. 그래서 초기 라벨링은 클래스당 20구간씩, 대표 구간만 먼저 구성해서 threshold validation을 꼼꼼히 할 계획입니다.</span>

## <span>지금 어디까지 왔나</span>

<span>현재는 핵심 파이프라인 구현을 진행 중입니다.</span>

- <span>✅ 전체 아키텍처 설계 완료</span>
- <span>✅ ffmpeg 기반 프레임 추출 (5초 구간, 10프레임)</span>
- <span>✅ CLIP 임베딩 추출 및 구간 평균</span>
- <span>✅ Logistic Regression 분류기 (dummy fallback 포함)</span>
- <span>▢ 행동 전환 기반 이벤트 감지 로직</span>
- <span>▢ </span><span><code>SessionLogger</code></span><span> 클래스 설계 (CSV + JSON 듀얼 출력)</span>
- <span>▢ tkinter GUI 연동</span>
- <span>▢ 실제 오리 데이터셋 라벨링 (클래스당 20구간 목표)</span>
- <span>▢ Pi 이식 및 오프라인 실행 검증</span>

<span>몇 가지 설계 결정을 공유하면:</span>

<span><strong>행동 전환 반응은 최소 3구간(15초) 유지 후에만 발생합니다.</strong></span><span> 5초 단위 분류가 짧게 흔들릴 수 있어서, 노이즈 반응을 막기 위한 smoothing 규칙입니다. (</span><span><code>MIN_STABLE_SEGMENTS = 3</code></span><span>으로 상수화)</span>

<span><strong>uncertain 처리도 신중하게 설계하는 중입니다.</strong></span><span> </span><span><code>CONFIDENCE_THRESHOLD = 0.55</code></span><span> 미만이면 uncertain으로 처리하는데, 연속 uncertain이 일정 횟수 이상 발생할 때만 별도 반응을 보여줍니다. 시연 중 uncertain이 너무 자주 뜨면 몰입감이 깨지기 때문입니다.</span>
